# 第3話【協働編】 補足ノート

note 本編「副業エンジニアのためのAIコーディング使い分けマップ【協働編】設計とレビューだけは人間」の補足素材です。新シリーズ `ai-coding-mapping-2026` の第 3 話、Claude Code 1 本で「設計 → 実装 → レビュー → PR」の 4 工程を分担する協業ワークフローの実走ログと再現手順を置きます。

## 本記事のスタンス開示

本編冒頭で 2 点を断っています。本サイトでも同じ前提です。

- **Claude Code 1 本に絞ります**: 第2話で扱った Claude Code ＋ Cursor のハイブリッド運用は本記事の射程外です。複数ツール比較ではなく、Claude Code 1 本で 4 工程をどう分担するかに集中しています。
- **題材は架空の SaaS 「タスクメッシュ」案件**: 本サイトの数値は、手元の個人プロジェクトで `tag_color_preference` 相当の構造のタスクを 3 本回した範囲で観察した実測です。タスクメッシュは構造を借りた架空案件で、実在の受託案件ではありません。

## 1 タスク題材

本編・本サイト共通の題材は次の 1 タスクです。

- **要件**: `User` モデルに `tag_color_preference` カラムを足し、ユーザーが自分のタグ色を保存できる API を 1 本追加する
- **エンドポイント**: `PATCH /users/me { tag_color_preference: string (hex 7 文字) }`
- **スタック**: Node.js 20 + Express + Knex + Postgres 16 + Vitest (本編で言及した最小構成)

## 収録コンテンツ

- [4 工程の実走ログ](./verification-log) — Claude Code に出した /plan / セルフレビュー / PR 下書き生成の **プロンプト全文**、返ってきた 6 ステップ計画、コミット 7 本の構造、セルフレビューが上げた未決事項 3 件、PR 下書き 24 行と人間が直した 3 行の before / after
- [読者再現手順](./reproduce) — 手元の Node + Postgres リポジトリで `tag_color_preference` 相当のタスクを Claude Code に投げて同じ 4 工程を回すための手順。マイグレーション DDL と最小ハンドラの雛形まで載せます

## 実測サマリ (本編の数値の出所)

本編 h2「コスト感と所要時間」の数値はこの範囲です。

| 指標 | 中央値 | レンジ |
|---|---|---|
| 設計 (人間、ノート 8 行) | 10 分 | 8〜12 分 |
| 実装 (Claude Code 自走) | 1 時間 50 分 | 1 時間 30 分〜2 時間 10 分 |
| レビュー (人間 + セルフレビュー 1 回) | 15 分 | 12〜18 分 |
| PR 下書き作成 + 人間が 3 行書き直し | 5 分 | 4〜8 分 |
| **合計実時間 (人間が手を動かした時間)** | **30 分前後** | 24〜38 分 |
| AI 自走時間 (Claude Code が回している時間) | 1 時間 50 分 | 1 時間 30 分〜2 時間 10 分 |
| コミット数 | 7 本 | 5〜8 本 |
| テスト pass 率 | 92% | 90〜95% |
| セルフレビューが上げた未決事項 | 3 件 | 2〜4 件 |
| そのうち人間が手を動かして判断した件数 | 1 件 | 1〜2 件 |

数値はすべて手元のリポで 3 本回した観察値で、スタック構成と Claude Code 側のバージョンにより前後します。
